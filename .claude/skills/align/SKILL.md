# align ā€” LLM skill guide

align is an architecture-conformance verification oracle: a fluent TS DSL compiles to a portable JSON IR,
a language plugin evaluates it against the real dependency graph on every call (always a fresh scan ā€” no
stale cache to distrust), and violations surface through the CLI and an MCP server. This guide is
generated by `align skill` from the installed binary's own live registries ā€” the rule-kind list, DSL verb
table, bullet grammar, gate list, and CLI command inventory below can never drift from what this specific
install actually supports.

## Rule kinds

Every rule kind the installed IR schema supports (walked live off `ruleIRSchema`):

- `arch.no-dependency` ā€” One component must not import from another. Authored via `arch.layer(x).cannotDependOn(y)` or `arch.component(x).isIsolated()`.
- `arch.no-cycles` ā€” No import cycle within a scope (component or the whole repo). Authored via `arch.noCycles(scope?, { includeTypeOnly? })`. Violations carry per-edge chain detail, not just file names.
- `arch.layers` ā€” A component may depend only on an explicit allowlist of other components. Authored via `arch.layer(x).canOnlyDependOn(...)`.
- `custom.host` ā€” An escape hatch referencing a `HostPredicate` your `align.config.ts` registers by name in its `hostRules` export (`Record<string, HostPredicate>`). REGISTRATION IS REQUIRED: `custom.host('foo')` in the ruleset with no `hostRules.foo` predicate hard-errors the gate at check time (it never silently reports green) ā€” register the predicate, fix the name, or remove the rule. Authored via `custom.host(hostRuleName)`.
- `arch.metric` ā€” A component-scoped numeric limit ā€” today only `metric: "loc"` (max lines per file) is promoted; fan-in/fan-out/instability remain reserved pending evidence. Authored via `arch.component(x).maxLinesPerFile(max)`.
- `security.manifest.source-hygiene` ā€” Flags any dependency specifier resolving to a git/http(s)/file/link source instead of the registry or `workspace:`. Authored via `security.manifest.sourceHygiene()`. Repo-wide ā€” no component scoping.
- `security.manifest.new-dependency` ā€” Fingerprints every current runtime/dev dependency at baseline time; a genuinely new dependency added afterward shows red until accepted. Authored via `security.manifest.newDependencyGate()`. Repo-wide ā€” no component scoping.

## DSL verbs

The authoring surface (`defineProject`'s `c.*` factories), introspected off the live builder objects:

| Verb | Meaning | Produces |
| --- | --- | --- |
| `arch.layer(x).canOnlyDependOn(...refs)` | x may depend only on the listed components ā€” any other outgoing dependency from x is a violation. | `arch.layers` |
| `arch.layer(x).cannotDependOn(...refs)` | x must not depend on any of the listed components ā€” one arch.no-dependency rule per listed component. | `arch.no-dependency` |
| `arch.component(x).isIsolated()` | no other component may depend on x, and x depends on none of them ā€” a no-dependency rule pair for every other component. | `arch.no-dependency` |
| `arch.component(x).maxLinesPerFile(max)` | every file classified into x must stay at or under max lines (metric: loc ā€” the only promoted arch.metric metric today). | `arch.metric` |
| `arch.noCycles(scope?, options?)` | no import cycles within scope (defaults to the whole repo). options.includeTypeOnly widens cycle detection to type-only edges (excluded by default). | `arch.no-cycles` |
| `custom.host(hostRuleName)` | references a HostPredicate registered under the same name in align.config.ts's sibling `hostRules` export (`Record<string, HostPredicate>`). Registration is required: an unregistered hostRuleName hard-errors the gate at check time (never silently reports green) ā€” register the predicate, fix the typo, or remove the rule. | `custom.host` |
| `security.manifest.sourceHygiene()` | any dependency specifier resolving to a git/http(s)/file/link source (not registry, not workspace:) is a violation. Takes no arguments ā€” the manifest scan domain has no notion of file-classified components. | `security.manifest.source-hygiene` |
| `security.manifest.newDependencyGate()` | every current runtime/dev dependency is fingerprinted at baseline time; only a genuinely new dependency added afterward shows red. Takes no arguments. | `security.manifest.new-dependency` |

## Tier-2 bullet grammar (doc authoring)

Sentence forms `- **Rule**: ...` bullets in an architecture doc parse into deterministically, zero LLM:

- `arch.no-dependency` ā€” <component> must not depend on <component>.
  e.g. `- **Rule**: api must not depend on ui.`
- `arch.layers` ā€” <component> may|can only depend on <component>[, <component> and/or <component>...].
  e.g. `- **Rule**: api may only depend on core.`
- `arch.no-cycles` ā€” no cycles. | no cycles within|in|for <scope>. | <scope> must have no cycles.
  e.g. `- **Rule**: no cycles.`
- `arch.metric` ā€” files in <component> must stay under <N> lines.
  e.g. `- **Rule**: files in core must stay under 500 lines.`
- `security.manifest.source-hygiene` ā€” dependency|dependencies (sources) must be (sourced from the) registry(-only).
  e.g. `- **Rule**: dependency sources must be registry-only.`
- `security.manifest.new-dependency` ā€” new dependency|dependencies requires|needs baseline approval|acceptance.
  e.g. `- **Rule**: new dependency requires baseline approval.`

## Doc authoring: the precision ladder

A markdown architecture/best-practices doc is a buildable intent source (`align build`, ADR 011) ā€” it
compiles to rules the way `package.json` resolves to a lockfile. `align build` (dry-run by default) and
the MCP `align_propose_rules` tool both compile a doc through the same **precision ladder**, most-trusted
form first ā€” always author at the highest tier your intent allows:

1. **Fenced ` ```align ` blocks compile verbatim, zero LLM.** Block content is a JSON `RuleFragment` ā€” the
   structural fields of one rule kind, minus `id`/`provenance` (both are always assigned by the build
   pipeline). Highest trust, most precise, most work to author by hand.
2. **Structured `- **Rule**: ...` bullets parse deterministically** ā€” see the generated bullet-grammar
   section above for the exact sentence forms. An LLM only grounds fuzzy component-name selectors against
   the components registry; it never invents rule structure.
3. **Free prose goes through two-pass clarification** (`align_propose_rules`, ADR 011): pass 1
   (Discovery) ā€” read the doc, output a short list of *concerns*, no IR yet, human confirms or skips each.
   Pass 2 (Refinement) ā€” IR is generated only for confirmed concerns, each selector grounded against
   component names (never raw paths), with a dry-run impact report before anything is written. This is the
   least-trusted, most-scaffolded path ā€” never skip straight to writing rules from prose without the
   human confirmation gate in between; that's the overwhelm failure mode this mode exists to prevent.

Nothing writes without an explicit `--apply` (CLI) or `apply: true` (MCP) ā€” the default is always a
dry-run diff + impact delta ("adds N new violations / masks M baselined"). A rule-level diff is minimized
(IR-identical rules keep their ids verbatim), so a prose typo fix re-proposing a section yields an empty
diff, not id churn.

## Fix-loop protocol

- **check ā†’ fix ā†’ re-check until green.** Run `align check` / `align_check` after every structural change. Fix what it reports, then re-check ā€” never assume a fix worked without a fresh verdict. Every check is a full fresh scan (ADR 005): there is no stale cache to distrust, so trust every verdict as current.
- **RED IS BLOCKING.** A red `align check` / `align_check` means the change is not done. Do not consider a structural change complete, hand control back, or move on to the next task while red.
- **Never edit align.config.ts or .align/** to force green.** Removing, weakening, or suppressing a rule in align.config.ts, or hand-editing anything under .align/ (baseline.json, generated-rules.json, rules.lock.json, cache), is not a fix ā€” it hides the finding instead of resolving it. Fix the code the rule is pointing at.
- **Baseline acceptance is a HUMAN decision.** `align baseline accept` / `align_baseline_accept` (gated off by default, ADR 006) tolerates a violation as existing debt ā€” it does not fix it. An agent under pressure to reach green must not self-serve this; propose it to the human instead of calling it unprompted.
- **Explain on demand, not by default.** Use `align explain <ruleId>` / `align_explain_rule` to understand WHY a rule fired ā€” its kind, `.because()` rationale, and constrained components (plus a Mermaid diagram for cycle/dependency violations) ā€” before proposing a fix, not as a first-pass survey of the whole ruleset.
- **Payloads are structured-fields-only ā€” render prose yourself if you need it.** Machine payloads (`align_check`, `align_violations`, `align check --json`) carry structured fields only (file, line, specifier, snippet, fixHint) ā€” no redundant prose `message` field (ADR 007, measured 3.6x token reduction). Passing gates report counts only, never per-item text.

## Baseline consent doctrine

The baseline tolerates existing debt so a new rule (or align's first run) on a mature repo does not
demand a wall of red on day one. Consent is explicit, never silent (ADR 006):

- `align init` seeds the baseline from the first full check. **Interactive**: prints a loud summary and
  asks. **Non-interactive/CI**: requires an explicit `--accept-existing` flag ā€” silence is never consent,
  and without the flag `align init` exits red.
- `align baseline accept [--rule <ruleId>]` accepts current violations (optionally scoped to one rule) ā€”
  a human-invoked CLI command.
- **MCP never self-serves baseline acceptance by default**: `align_baseline_accept` is gated behind
  `allowBaselineFromMcp`, default `false`. An agent cannot grant itself amnesty from a rule it is failing.
  If a violation looks like it should be baselined as pre-existing debt rather than fixed, say so to the
  human and let them run the acceptance ā€” don't call an acceptance tool unprompted under pressure to turn
  a red verdict green.

## MCP payload & tool reference

`align mcp` (stdio) exposes:

- **`align_check`** ā€” fresh full scan, green/red/error verdict + per-gate counts. No arguments.
- **`align_violations`** ā€” current violations in priority order (architecture > security > types > lint >
  format), structured fields only, paginated (`cursor`).
- **`align_explain_rule`** ā€” one rule's kind, `.because()` rationale, constrained components with example
  files, and a Mermaid diagram for cycle/dependency violations. Pull-on-demand only ā€” never inlined into
  `align_check`/`align_violations` payloads.
- **`align_propose_rules`** ā€” two-pass doc-to-rules compilation (see the doc-authoring section above).

Payload discipline applies to every tool above (ADR 007): passing gates are counts, never per-item text;
no `message` prose field duplicating structured fields (`fromFile`/`toFile`/`specifier`/`line`/`snippet`
already exist structurally ā€” render prose yourself if you need it); results are priority-sorted before
pagination/truncation; caps and pagination are mandatory, not optional. A red `align_check` response for a
handful of violations costs roughly 1K tokens, not tens of thousands.

## CLI commands

- `align init [options]` ā€” Detect components, write a starter align.config.ts, and seed the baseline.
- `align check [options]` ā€” Run architecture rules against a fresh scan of the repo. Exit 0 iff green.
- `align baseline [options] [command]` ā€” Manage the violation baseline (tolerated debt).
  - `align baseline accept [options]` ā€” Accept current violations into the baseline (optionally scoped to one rule).
  - `align baseline prune [options]` ā€” Remove baseline entries for violations that no longer exist; report moved entries.
  - `align baseline show [options]` ā€” List baselined violations (optionally scoped to one rule).
- `align explain [options] <ruleId>` ā€” Explain one architecture rule: its kind, rationale, and constrained components.
- `align doctor [options]` ā€” Read-only advisory survey: dead tsconfig path aliases, uncertainty breakdown, unmapped files, workspace-orphaned packages, empty components. Never fails ā€” exit code is always 0.
- `align build [options]` ā€” Compile a markdown architecture doc into the ruleset (ADR 011): fenced ```align blocks + structured `- **Rule**:` bullets, zero LLM. Default is dry-run (prints the proposal diff + impact delta, writes nothing).
- `align agent [options] [command]` ā€” Built-in BYOK LLM fix loop (Stage 4, ADR 010). Requires ANTHROPIC_API_KEY.
  - `align agent run [options]` ā€” DISCOVER -> GROUP -> PLAN+FIX -> APPLY -> VERIFY -> REPAIR -> ESCALATE -> DONE -> TERMINAL MERGE. Refuses a dirty worktree; every apply is a commit on a fresh align/fixes-<date> branch; never touches align.config.ts or .align/**.
- `align mcp [options]` ā€” Start the align MCP server (stdio) exposing align_check / align_violations / align_explain_rule.
- `align skill [options]` ā€” Print the LLM-facing align authoring/fixing guide (rule kinds, DSL verbs, bullet grammar, gates, and CLI commands generated live from the installed binary ā€” never hand-written prose that can drift). --install writes .claude/skills/align/SKILL.md into this repo.
